User Valued Work#

We often need to balance prioritization between internal-facing work and user-facing work. Itâ€™s helpful to distinguish between these.

User-valued work#

This is work that directly affects users. It solves their pain and gives them a delightful experience. It includes work like the following:

  1. Features that solve user pain

  2. Features that delight users

  3. Documentation that many people read and engage with

  4. Directly answering user questions

Developer-valued work#

This is work that mostly developers see.

  1. Paying down technical debt

  2. Fixing CI

  3. Writing well thought out technical documentation, even if it isnâ€™t widely read

  4. Discovering bugs or problems in the system

  5. Writing design documents

  6. Developing features behind a feature flag

  7. Developing features but not releasing them or releasing them only very recently so that most users donâ€™t have them yet

Developer valued work is also valuable, but only in that it makes us faster in delivering user-valued work to users.

Timing and Judgement#

We are judged as a team on user-valued work in the long term. Developer-valued work is good, but only in service of user-valued work. Itâ€™s great to do developer-valued work for a week or a month, but beyond that it comes into question.

In a professional context when upper-management judges the performance of a team, user-valued changes are typically what matter. This typically matches how the public judges the work as well.
